BLOOD DONATION PLATFORM - SOURCE OF TRUTH

Purpose:
This document defines the current functional behavior and technical constraints for the live design in this repository.


SECTION A - CURRENT FUNCTIONAL REQUIREMENTS

FR-01 System Boundaries
1. WHO and Redcross are simulated external source systems.
2. ETL is the only component allowed to consume WHO and Redcross APIs for ingestion.
3. Frontend and backend business workflows must not depend directly on WHO or Redcross APIs.

FR-02 Data Provisioning
1. Seeder generates deterministic dummy source-style data.
2. Seeder overwrites and regenerates `db-seeder/generated_seed_preview.csv` each run.

FR-03 ETL Bulk Load
1. Bulk load is manual (not automatic at ETL startup).
2. Bulk load trigger endpoint is `GET /admin/etl/bulk-load` on ETL service.
3. Bulk load starts async and returns accepted response text.
4. Re-trigger after completion must be idempotent (`bulk load already completed`).

FR-04 ETL Incremental
1. Incremental ETL runs by cron scheduler.
2. Default schedule is `0 0 0 * * *` in `Asia/Kolkata`.
3. Incremental runs only after bulk completion state is true.
4. Incremental updates operational projections and analytics facts through ETL pipeline processing.

FR-05 Blood Bank Search (Anonymous)
1. Blood-bank search is available without donor login.
2. Backend supports pincode-driven nearest lookup.
3. Query behavior uses geo-distance sorting and practical result limiting.

FR-06 Donor Login and Auth Gating
1. Donor actions are gated behind OTP flow.
2. Local OTP behavior is deterministic (`123456`).
3. On successful OTP verification, user identity is resolved or created in backend.

FR-07 Donor Search (Authenticated)
1. Donor search requires authenticated user context in UI flow.
2. Matching uses blood group compatibility and geo-aware search.
3. UI displays aggregate distance bucket counts.

FR-08 Request Lifecycle
1. User can create request from donor search context.
2. User can view request history and response history.
3. User can trigger re-request preview, re-request, and dispatch-next for eligible requests.
4. Active request guard is enforced per user.

FR-09 Donor Response Update
1. Dummy endpoint exists to mark donor response by phone number input.
2. Response is linked to request context and visible in user response history.

FR-10 Logging
1. API entry and exit logs are controller-level and minimal.
2. Exception handling is centralized with structured responses.

FR-11 Analytics and Observability
1. Grafana must load with provisioned Elasticsearch and ClickHouse datasources.
2. Simple dashboard panels must execute queries successfully against both datasources.


SECTION B - TECHNICAL CONSTRAINTS

TC-01 Storage Roles
1. PostgreSQL primary stores users, searches, requests, and responses.
2. Elasticsearch stores operational search projections.
3. ClickHouse stores analytics and ingestion facts.

TC-02 Source Isolation
1. Backend must not embed WHO or Redcross extraction logic.
2. Frontend must not call WHO or Redcross APIs for app workflows.

TC-03 API Contract Discipline
1. Public API request and response payloads remain typed DTOs.
2. Error responses remain typed and predictable.

TC-04 ETL Structure
1. ETL orchestration stays service-oriented (no god-class design).
2. Bulk and incremental behavior must remain explicit and deterministic.
3. ETL state persistence uses `/data/state.json` for sync continuation.

TC-05 Runtime Predictability
1. Input validation is fail-fast.
2. Fresh environment runs should behave deterministically.

TC-06 Scope Simplicity
1. No JWT or complex auth in current scope.
2. Keep implementation minimal and avoid unnecessary abstraction.

1. Start stack with clean build: `docker compose up -d --build`
2. Verify all critical containers are running/healthy: `docker compose ps`
3. Seed source data: `python db-seeder/seed.py`

5. Re-trigger ETL bulk and verify idempotent completion response

9. Validate backend `GET /blood-banks/search`

12. Validate Grafana login on `http://localhost:3000`

SECTION F - METADATA STATUS

Current metadata already present in the repository:
1. Source schema metadata: PostgreSQL schema files for backend, WHO, and Redcross.
2. Warehouse schema metadata: ClickHouse DDL in `Clickhouse/resources/schema_clickhouse.sql`.
3. ETL load state metadata: `etl-service/src/main/java/etl/service/EtlStateStore.java` persists last sync timestamps in `/data/state.json`.
4. Operational datasource metadata: Grafana provisioning files for Elasticsearch and ClickHouse.
5. Documentation metadata: `README.md` and this file describe runtime topology and table inventory.

Implemented warehouse metadata catalog tables:
1. `blood_ops.meta_source_system`
2. `blood_ops.meta_dataset`
3. `blood_ops.meta_column`
4. `blood_ops.meta_lineage`
5. `blood_ops.meta_load_audit`

What is not yet present:
1. Automatic population of the metadata catalog tables from the ETL pipeline.
2. A UI or admin API for browsing metadata catalog contents.

Recommended metadata layer for this project:
1. `meta_source_system` - source system code, name, owner, active flag.
2. `meta_dataset` - logical dataset name, physical table, engine, refresh mode.
3. `meta_column` - dataset, column name, data type, business meaning, nullable flag.
4. `meta_lineage` - target column, source column, transform rule, source system.
5. `meta_load_audit` - batch_id, source system, target, row counts, started_at, ended_at, status, message.
