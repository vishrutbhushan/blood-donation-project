BLOOD DONATION PLATFORM - SOURCE OF TRUTH

Purpose:
This document defines the required functional behavior (FRs) and the mandatory technical constraints/restrictions for the project.


SECTION A - FUNCTIONAL REQUIREMENTS (FR)

FR-01 System Boundaries
1. WHO and Redcross are simulated external source systems.
2. ETL is the only component allowed to consume WHO and Redcross APIs.
3. Backend and frontend user flows must not directly depend on WHO/Redcross APIs.

FR-02 Data Provisioning
1. Seeder must generate deterministic dummy source-style data.
2. Seeder must overwrite and regenerate a CSV preview file for test validation.

FR-03 ETL Bulk Load
1. ETL bulk load is manually triggered (not automatic at startup).
2. ETL bulk load must ingest both blood bank and donor datasets.
3. ETL must normalize and write data into Elasticsearch and ClickHouse.
4. Re-triggering completed bulk load must be idempotent.

FR-04 ETL Incremental Load
1. ETL incremental sync runs on schedule after bulk completion.
2. Incremental sync fetches only changed records from source systems.
3. Incremental sync upserts updates into Elasticsearch and ClickHouse.

FR-05 Blood Bank Search (Anonymous)
1. Blood-bank search must be usable without donor login.
2. Backend must support pincode-driven nearest blood bank lookup.
3. Query behavior must use geo-distance and return nearest 20 records max.
4. UI must display search results from backend APIs only.

FR-06 Donor Login
1. Donor tab is visible/usable only for logged-in donors.
2. Current local OTP behavior is deterministic and hardcoded.
3. On successful OTP verification, user identity is resolved/created in backend.

FR-07 Donor Search (Authenticated)
1. Donor search requires authenticated donor context.
2. Donor matching uses blood group and pincode with geo-aware querying.
3. UI must not reveal donor personal details during search stage.
4. UI must display counts by distance buckets:
   a. Less than 10 km
   b. Less than 50 km
   c. Greater than 50 km

FR-08 Request Creation
1. User can create a blood request from donor search context.
2. Search context must be persisted.
3. Initial dispatch targets nearest 20 eligible donors.
4. Contacted donor entries are persisted for tracking.
5. Donor personal details remain hidden to requester unless donor has responded.

FR-09 Re-request
1. User can create a follow-up request from an eligible prior request.
2. Re-request must show the same distance bucket counts before confirmation.
3. Re-request dispatch must avoid overlap with previously contacted donors for that request chain.

FR-10 Dispatch Next
1. User can trigger "next 20" for an active request.
2. System advances dispatch window and contacts next eligible donors.
3. Contact counters and timestamps must update.

FR-11 Donor Response Update
1. A dummy API must exist to mark donor responded=true using patient mobile number input.
2. Response must be linked to request and user context.
3. Requester can view list of responded donors with related request ID.

FR-12 Active Request Guard
1. System must enforce one active request guard per user.
2. New request creation is blocked when active request exists.

FR-13 Request and Response History
1. User can view request history.
2. User can view donor response history.
3. UI must expose actions such as re-request and dispatch-next based on eligibility.

FR-14 Analytics
1. Grafana must read Elasticsearch and ClickHouse datasources.
2. Dashboards must reflect ETL-loaded and ETL-incremented data.

FR-15 Error and Validation Behavior
1. Invalid inputs must fail fast with typed API errors.
2. Validation errors must return structured validation payloads.
3. Unexpected exceptions are centrally handled with safe response payloads.

FR-16 Logging Behavior
1. Logs should be minimal and controller-focused.
2. Log important API triggers/results and centralized exceptions.
3. Avoid service-level request logs except ETL trigger visibility where needed.


SECTION B - TECHNICAL CONSTRAINTS AND RESTRICTIONS

TC-01 Storage Responsibility
1. PostgreSQL is the system of record for users, searches, requests, responses.
2. Elasticsearch is the operational search store for blood bank and donor queries.
3. ClickHouse is analytics-only (reporting/Grafana), not operational request search.

TC-02 Data Source Isolation
1. Backend must not embed source extraction logic for WHO/Redcross.
2. Frontend must not call WHO/Redcross APIs for application workflows.

TC-03 Geo and Indexing Constraints
1. Elasticsearch bank/donor indexes must keep location as geo_point.
2. Nearest blood-bank query must use Elasticsearch native geo sort and max size 20.
3. ETL must resolve pincode to coordinates and continue safely on missing mappings using fallback defaults.

TC-04 ETL Startup and Mapping
1. ETL must ensure Elasticsearch templates/mappings exist before indexing begins.
2. ETL must support bulk-load idempotency and incremental continuation.

TC-05 API Contract Constraints
1. External/public API payloads must be typed DTO contracts.
2. No Map-based request/response contracts for public APIs.
3. Error payloads must remain typed DTO responses.

TC-06 Internal Code Constraints
1. Method signatures should prefer typed objects over generic maps.
2. Internal ETL processing may use in-memory maps for merge/index mechanics only.
3. Redcross and WHO SQL queries must be class-level final constants.
4. Redcross and WHO DTO boilerplate must use Lombok conventions.
5. Elasticsearch integration should use class-level query templates plus typed response mapping.

TC-07 Security and Simplicity Constraints (Current Scope)
1. Do not add JWT or complex auth for this dummy project scope.
2. Keep login gating simple: donor flows hidden/blocked when not logged in.
3. Current local OTP stays deterministic/hardcoded.

TC-08 Behavioral Quality Constraints
1. Input validation must be deterministic and fail-fast.
2. Runtime behavior must preserve current-path predictability for fresh environment runs.
